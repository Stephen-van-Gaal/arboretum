# sync-contracts — `sync-contracts.sh` contracts.yaml Producer Contract

The seam between `scripts/sync-contracts.sh` (the auto-generator of `contracts.yaml` — the version-pin manifest derived from each governed spec's `## Requires` / `## Provides` definition references) and its downstream consumers, chiefly `scripts/validate-cross-refs.sh` (Check 3 reads `contracts.yaml` back and compares it against the live spec tables). This contract pins the `contracts.yaml` shape so a generator format change can never silently desync from the validator that parses it — the exact #124-class coupling guard the contract pattern exists to enforce.

## Producer

`scripts/sync-contracts.sh` — producer-type: `script`.

Scans every `docs/specs/*.spec.md` under the project directory. For each spec it extracts the `## Requires` and `## Provides` sections (`sed -n '/^## Requires/,/^## /p'`) and pulls out every `definitions/<path>@v<N>` definition-version pin (`grep -oE 'definitions/[^@|[:space:]]+@v[0-9]+'`). Specs with no definition references are skipped entirely (they do not appear in the output tree). Pins are deduplicated and sorted (`sort -u`).

It then assembles a `contracts.yaml` document: a header comment block, a literal top-level `seam-contracts: docs/contracts/` key (preserved verbatim across regenerations — the generator only derives the `specs:` tree, never the per-file seam contracts), and a `specs:` mapping. Under `specs:`, each spec that has at least one pin becomes a key (`<short-name>:`, the basename with the `.spec` suffix stripped), with optional `requires:` and `provides:` sub-maps whose entries are `<def-path>: v<N>`. When no spec has any pin, `specs:` is emitted as `specs: {}`.

Path resolution uses a positional `[project-dir]` arg (defaults to `pwd`). With `--dry-run` the document is printed to stdout and **no file is written**; otherwise it is written to `<project-dir>/contracts.yaml` and a `Updated <path>` line is printed.

The script runs under `set -euo pipefail` and exits non-zero (with a stderr diagnostic) when the specs directory is absent or contains no `*.spec.md` files. It is a non-atomic single-shot `> "$CONTRACTS"` redirect — there is no mktemp + mv (unlike the cache producers); the file is small and regenerated wholesale, and the only consumers are synchronous CI scripts, never a concurrent session-start path.

## Consumer

Consumer-type: `script`. Two downstream readers of the generated `contracts.yaml`:

- **`scripts/validate-cross-refs.sh`** (script). Check 3 (`━━━ Check 3: contracts.yaml matches spec tables ━━━`) reads each spec's `specs:` section out of `contracts.yaml` (`sed -n "/^  ${short_name}:/,/^  [^ ]/p"`), parses the `requires:` / `provides:` entries (`grep -oE 'definitions/[^:[:space:]]+: *v[0-9]+'`, normalized to `<path>@v<N>`), and bidirectionally compares them against the live spec tables. A format change to the indentation, the `<path>: v<N>` shape, or the per-spec key naming breaks this parse.
- **`.claude/hooks/session-start.sh`** (hook). Treats `contracts.yaml` as a required project artifact: its absence is reported in the boot-banner "missing infrastructure" list, and its definition-version pins are parsed for the `[Stale Version Pins]` drift block.

**Consumer obligations:**

- The consumer MUST parse the two-space-indented `specs:` tree: spec keys at 2-space indent, `requires:` / `provides:` at 4-space indent, individual `<def-path>: v<N>` pins at 6-space indent.
- The consumer MUST tolerate the literal `seam-contracts: docs/contracts/` top-level key and MUST NOT mistake it for a spec entry.
- The consumer MUST tolerate `specs: {}` (no spec carried any pin) without erroring.
- The consumer MUST NOT require a spec with zero definition references to appear in the tree.

## Protocol shape

### Inputs

`scripts/sync-contracts.sh` accepts:

- **`[project-dir]`** — positional, defaults to `pwd`. Root under which `docs/specs/` is scanned and `contracts.yaml` is written.
- **`--dry-run`** — flag. Print the document to stdout; write nothing.

Reads:

- `<project-dir>/docs/specs/*.spec.md` — the `## Requires` and `## Provides` sections of each, for `definitions/<path>@v<N>` pins.

### Outputs

Writes `<project-dir>/contracts.yaml` (unless `--dry-run`). Document shape:

```yaml
# contracts.yaml — version pins for contract test enforcement
# Auto-generated by sync-contracts.sh — <ISO-8601 UTC>
# … header comment block …

seam-contracts: docs/contracts/

specs:
  <spec-short-name>:
    requires:
      definitions/<path>.md: v<N>
      …
    provides:
      definitions/<path>.md: v<N>
      …
  …
```

- `seam-contracts: docs/contracts/` is always present, verbatim, as a literal top-level key.
- `specs:` is always present. When no spec carries a pin it is `specs: {}`.
- Each spec key is the spec basename with `.spec` stripped, at 2-space indent.
- `requires:` / `provides:` are emitted only when that spec has at least one pin of that kind, at 4-space indent.
- Pin lines are `<def-path>: v<N>` at 6-space indent, deduplicated and sorted.

Exit codes:

- `0` — document generated (written, or printed under `--dry-run`).
- `1` (non-zero) — specs directory absent, or no `*.spec.md` files present. Stderr diagnostic.

### Invariants

- **Literal seam-contracts key.** `contracts.yaml` always contains the line `seam-contracts: docs/contracts/` as a top-level key. The generator only derives the `specs:` tree; it never reads or rewrites the per-file seam contracts under `docs/contracts/`.
- **Output structure.** The document has exactly two top-level keys: `seam-contracts` and `specs`. Adding or removing a top-level key is a contract change requiring a coordinated consumer update.
- **Indentation contract.** Spec keys at 2 spaces, `requires:`/`provides:` at 4 spaces, `<path>: v<N>` pins at 6 spaces. `validate-cross-refs.sh` Check 3 depends on this exact nesting.
- **Pin shape.** Each pin is `definitions/<path>.md: v<N>` — a YAML-mapping `key: value`, NOT the `@v<N>` form used inside spec tables. The version is `v` followed by one or more digits.
- **Skip-empty-specs.** A spec with no `definitions/…@v<N>` reference in either section does not appear in the `specs:` tree at all.
- **Empty-tree form.** When no spec carries any pin, `specs:` is emitted as `specs: {}` (single line), never as a dangling `specs:` with no children.
- **Dedup + sort.** Pins within a `requires:` or `provides:` block are deduplicated and sorted (`sort -u`).
- **Dry-run never writes.** `--dry-run` prints to stdout and leaves `contracts.yaml` untouched (mtime + content unchanged).
- **Round-trip with validator.** A `contracts.yaml` freshly generated from a set of specs MUST pass `validate-cross-refs.sh` Check 3 against those same specs with zero Check-3 issues — the generator and the validator parse the identical schema.

## Test surface

- **SC-1:** Generates a `contracts.yaml` containing the literal `seam-contracts: docs/contracts/` top-level key and a `specs:` key.
- **SC-2:** A spec with a `## Requires` pin (`definitions/foo.md@v1`) produces a `<spec>:` entry with a `requires:` sub-map carrying `definitions/foo.md: v1` at 6-space indent.
- **SC-3:** A spec with a `## Provides` pin produces a `provides:` sub-map with the `<path>: v<N>` line.
- **SC-4:** A spec with no definition references does NOT appear in the `specs:` tree.
- **SC-5:** When no spec carries any pin, the output contains `specs: {}`.
- **SC-6:** `--dry-run` prints the document to stdout and writes no `contracts.yaml` (file absent / unchanged).
- **SC-7:** Round-trip: a generated `contracts.yaml` passes `validate-cross-refs.sh` Check 3 against the same specs with zero Check-3 issues, including specs that carry both `requires:` and `provides:` blocks.
- **SC-8:** Absent specs directory → exit non-zero with a stderr diagnostic.
